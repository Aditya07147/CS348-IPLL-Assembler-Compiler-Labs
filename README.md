##  Theoretical Background & Compiler Architecture (Dragon Book Foundations)

As established in Aho, Lam, Sethi, and Ullman's classic textbook, *Compilers: Principles, Techniques, and Tools* (The Dragon Book), a compiler operates as a series of phases, commonly divided into a **Front-End** (analysis) and a **Back-End** (synthesis). This project implements a complete compiler front-end pipeline for the nanoC language:

$$\text{Source Code} \xrightarrow{\text{Lexical Analyzer (Flex)}} \text{Token Stream} \xrightarrow{\text{Syntax Analyzer (Bison)}} \text{Parse Tree / Syntax Tree}$$

```text
                               +-------------------+
                               |    Source Code    |
                               +---------+---------+
                                         |
                                         v (Character Stream)
                               +---------+---------+
                               |  Lexical Analyzer | <-----+
                               |      (Flex)       |       | (Symbol Table
                               +---------+---------+       |  Queries/Updates)
                                         |                 |
                                         v (Token Stream)  v
                               +---------+---------+   +---+---+
                               |  Syntax Analyzer  | < |Symbol |
                               |     (Bison)       |   | Table |
                               +---------+---------+   +-------+
                                         |
                                         v (Parse/Syntax Tree)
                               +---------+---------+
                               | Semantic Analyzer |
                               +-------------------+
```

### 1. Lexical Analysis (Flex)
The lexical analyzer (scanner) is the first phase of the compiler. Its primary role is to read the input character stream, group characters into meaningful sequences called *lexemes*, and output a stream of *tokens*. 
*   **Mathematical Foundation**: Under the hood, Flex compiles the regular expressions defined in `lexer_nanoc.l` into regular definitions. These are translated into a **Non-deterministic Finite Automaton (NFA)** and ultimately optimized into a **Deterministic Finite Automaton (DFA)**. 
*   **Symbol Table Integration**: As tokens are scanned, identifiers are logged into a dynamic symbol table alongside scope depth and type descriptors, guaranteeing $O(1)$ lookup and insertion time during compilation.

### 2. Syntax Analysis (Bison)
The syntax analyzer (parser) takes the token stream from the lexer and verifies that the string can be generated by the grammar of the source language.
*   **Parsing Strategy**: Bison constructs a **Look-Ahead LR (LALR(1)) parsing table** to drive an efficient bottom-up, shift-reduce parsing algorithm. It maintains a state stack and evaluates syntactic correctness by performing *shifts* (pushing tokens onto the stack) and *reductions* (replacing stack elements with a non-terminal based on a production rule).
*   **Conflict Resolution**: Natural language ambiguities—such as the classic **dangling-else** shift-reduce conflict—are explicitly resolved within `parser_nanoc.y` using Bison's operator precedence and associativity declarations.

### 3. Assembler Pass Architectures
An assembler is a translator that converts assembly mnemonics into relocatable machine-readable object code.
*   **One-Pass Assembler**: Resolves forward reference address leaks dynamically on-the-fly by building a linked-list backpatching structure. When a forward-referenced symbol is finally defined, the assembler traverses its reference chain and fills in the resolved absolute memory address directly.
*   **Two-Pass Assembler**: Separates representation from resolution. Pass 1 acts as a "metadata collector," scanning instructions to assign memory locations (Location Counter) and populate the Symbol Table (`SYMTAB`). Pass 2 performs translation, mapping symbols to absolute addresses and opcodes to binary representation before emitting formatted object records.

##  Assignment Modules & Implementation Details

###  Assignment 1: x86-64 Assembly Systems (NASM)
Implements four low-level tasks targeting bare-metal execution on the Linux x86-64 architecture:
*   **Fibonacci Generation**: Calculates and prints Fibonacci numbers dynamically, utilizing loop instructions and explicit hardware carry/overflow flag verification to halt before register wrapping.
*   **File Stream Processing (`set_a_q11.asm`)**: Opens, reads, and processes character inputs from a file stream (`inputa.txt`) using system calls (`sys_open`, `sys_read`, `sys_write`), handling EOF and file descriptor allocation.
*   **Interactive K-th Largest Float (`set_b_q10.asm`)**: Reads floating-point parameters, manages memory arrays, and computes order statistics to output the k-th largest element.
*   **Graph Cycle Detection (`set_b_q2.asm`)**: Implements cycle detection on a static directed graph utilizing a Depth-First Search (DFS) traversal algorithm via recursive stack manipulation.

---

###  Assignment 2: SIC Assembler Architectures
Implements and compares the two standard assembler architectures in C for the **Simplified Instructional Computer (SIC)**:
1.  **One-Pass Assembler**: Resolves forward references on-the-fly without requiring an intermediate file. It processes input assembly code sequentially and utilizes a linked-list-based backpatching mechanism to write address resolutions once operands are defined.
2.  **Two-Pass Assembler**:
    *   **Pass 1**: Allocates addresses to labels, computes the Location Counter (`LOCCTR`), and builds the Symbol Table (`SYMTAB`).
    *   **Pass 2**: Resolves instruction opcodes using `OPTAB`, translates labels to absolute addresses, and emits final hexadecimal Object Code (Header, Text, and End records).

---

###  Assignment 3: nanoC Lexical Analyzer (Flex)
A robust lexical scanner for **nanoC** utilizing Flex. It processes the input program (`test.nc`), extracts regular expressions for programming tokens, and outputs:
*   **`tokens.txt`**: A serialized stream of classified tokens (keywords, identifiers, arithmetic/relational operators, and constants).
*   **`symbol_table.txt`**: A dynamically maintained symbol table logging identifier scopes, types, and byte offsets.

---

###  Assignment 4: nanoC Syntax Analyzer (Flex & Bison)
An end-to-end syntactic parser constructed by combining **Flex** (lexer) and **Bison** (LALR(1) parser). 
*   Interprets the Context-Free Grammar (CFG) of the nanoC programming language.
*   Parses core constructs including nested conditional structures (`if-else`), loops (`for`, `while`), assignment operations, variable declarations, and function signatures.
*   Performs structural syntax validation, printing syntactic trees and tracking grammar reduction states to report syntax errors in the program structure.
